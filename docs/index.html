关键洞察

openEuler 开发者工具集中度最高：developer-experience 类 114 个（占该社区的 57.3%），skills（85）和 DevStore（64）是核心产出仓库。与 CANN/Ascend 的测试-构建-工具链三角结构截然不同，openEuler 的基础设施高度聚焦于开发者 CLI、包管理和日志分析等上层工具。
CANN 新增 other-infra 类别 21 个：本轮分类新增较多部署类（6）和无法归入现有子类的其他基础设施问题，反映出 CANN 社区基础设施问题的多样性在增加，分类体系可能需要进一步细化。
CANN vs Ascend 结构差异持续：CANN 以 testing-infra（57）、build（44）、code-quality（24）为主，偏底层编译测试；Ascend 以 ci/cd（18）、testing-infra（16）、build（12）为主，CI 流水线占比显著更高（22.8% vs 5.5%），两社区研发重心差异明显。
BoostKit 仍是唯一零基础设施关注的社区：19 个 issue 覆盖 8 个仓库，全部为功能需求、CVE 漏洞和文档反馈。该社区持续缺乏 CI/测试/构建类输入，基础设施问题识别机制可能尚未建立。
ci/cd 与 testing-infra 是跨社区共性基础设施支柱：ci/cd 全社区合计 71 个，testing-infra 合计 77 个，两者占总基础设施类的 28.1%。openEuler 贡献了 ci/cd 的 47.9%（34/71），CANN 贡献了 testing-infra 的 74.0%（57/77）。
